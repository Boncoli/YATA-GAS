# YATA Project Guide (AI Context Document)

このドキュメントは、**YATA (Yet Another Trend Analyzer)** プロジェクトの全体像、アーキテクチャ、および開発・運用の「虎の巻」です。
AIエージェントや開発者は、作業を開始する前に**必ずこのファイルを読み込み、コンテキストを把握してください。**

---

## 1. プロジェクト概要

**YATA** は、元々Google Apps Script (GAS) で開発された「AI駆動型ニュース収集・分析プラットフォーム」を、**ローカルNode.js環境 (Raspberry Pi等)** に移植したプロジェクトです。

### コア・コンセプト
*   **GAS互換性**: メインロジック (`lib/YATA.js`) はGAS版とほぼ同一のコードベースを維持。
*   **GAS Bridge**: `lib/gas-bridge.js` がGASの標準サービス (`PropertiesService`, `UrlFetchApp`, `SpreadsheetApp`) を模倣し、SQLiteやローカルFSに処理を委譲。
*   **RAMディスク運用**: SDカード保護と高速化のため、実行時にDBをメモリ (`/dev/shm`) に展開して処理。
*   **ハイブリッド構成**: ロジックは Node.js、可視化（ダッシュボード）は Python。

## 3. データ運用とメンテナンス (重要)

### データベース (SQLite)
*   **ファイル**: `yata.db` (メイン), `/dev/shm/yata.db` (実行時のRAMコピー)
*   **WALモード対応 (2026/02 強化)**: 
    *   高速書き込みを実現するため WAL (Write-Ahead Logging) を使用。
    *   `run-ram.sh` により、本体 (`.db`) だけでなく差分ログ (`.db-wal`) および共有メモリ (`.db-shm`) も同期対象とし、データの完全な整合性を確保。
*   **主要テーブル**:
    *   `collect`: ニュース記事、要約、ベクトルデータ（通常ベクトル＋手法ベクトル）。
    *   `drive_logs`: [New] iPhoneからのライフログ（位置、高度、バッテリー、メモ）。
    *   `weather_forecast`: 天気予報。
    *   `log`: システムログ。
    *   `fuel_logs`: [New] 愛車 (CX-80) の燃費・給油記録。
    *   `daily_health`: [New] 1日1行の統合ヘルスケアログ（歩数、睡眠時間、HRV、安静時心拍、消費カロリー、睡眠詳細JSON）。

### アーカイブ・DB最適化戦略 (2026/03 強化)
DBの肥大化を防ぐため、古い記事のアーカイブ化と、物理的なサイズ削減（VACUUM）を組み合わせて運用しています。

1.  **アーカイブ実施 (半自動)**:
    *   手順: `tasks/archive-old-articles.js` が動的に180日前を計算し、JSONへ退避後に `VACUUM` を実行。
    *   **実績**: 2026年2月11日、アーカイブ残存データの削除と VACUUM により **217MB -> 32MB** への削減を確認。
    *   **2026年3月24日**: 最新モデル `gpt-5-nano` への切り替えとバッチ処理の安定化。詰め込みパッチを無効化し、1記事1リクエストの並列処理に戻すことで、nanoモデルでの構造化要約の確実性を向上。
    *   **重要知見 (2026/03/22)**: `DELETE` 実行だけではファイルサイズは減らず、「空き地」としてファイル内に残り続ける。`VACUUM` を実行して初めて物理的に圧縮され、OSに容量が返却される。今回、170MBから85MBへの急減は、溜まっていた「空き地」が手動または自動の `VACUUM` で返還されたことによる正常な挙動。

2.  **毎晩の自動最適化 (2026/03/22 導入)**:
    *   **頻度**: 毎日 04:35 (バックアップ時)
    *   **内容**: `do-backup.sh` 内で `clean-db-duplicates.js` を実行。`collect` テーブルの重複を掃除した直後に **RAM上で `VACUUM` を強制実行**。
    *   **効果**: SDカードへの書き込み量を最小化しつつ、NASへ送るバックアップファイルを常に最小サイズに保つ「黄金ルート」を確立。

### 定期メンテナンス (自動)
*   **頻度**: 毎月1日 04:02 (crontab設定済み)
*   **実行ファイル**: `tasks/maintenance-db.js`
*   **対象**: 各種ログテーブル (`weather_log`, `remo_log`, `finance_log` 等) を180日分保持してアーカイブ・削除。
*   **注意**: 毎晩の `do-backup.sh` での掃除に加え、月次でさらに深いクリーンアップを行う二段構えの運用。

### NASへの自動バックアップ (2026/03 最適化)
SDカードの故障に備え、毎日 04:35 に NAS への包括的なバックアップを実施。2026/03より、転送前にDBを最適化するフローへ改善。
*   **スクリプト**: `maintenance/do-backup.sh`
*   **フロー**: **RAM上で最適化(VACUUM) -> SDへ同期 -> NASへ転送 -> 再起動** の順序で実行。
*   **バックアップ対象**:
    *   **システム設定**: `/etc` フォルダ全体を `etc_backup.tar.gz` として圧縮保存（リンク保持のため）。
    *   **ホームディレクトリ**: `/home/boncoli` 以下を `rsync` 同期（`.git` 履歴を含む）。
    *   **データベース**: `yata.db` の日付付きスナップショット（過去30日分）。
    *   **ログファイル**: RAMディスク上のログを NAS (`/mnt/nas/yata_logs/`) へ日付付きで退避（過去30日分）。
    *   **復旧情報**: `package_list.txt` (インストール済パッケージ), `crontab_last.txt` (cron設定)。

### ログの出力先と永続化戦略 (2026/03 新設)
システム全体のログ（出力）は、**「SDカードの寿命保護（I/O削減）」**と**「トラブル時の証拠保全」**という相反する要件を両立するため、以下の3層構造（Tiers）で厳格に管理されています。

#### 1. 短期・揮発性ログ（RAMディスク）
*   **場所**: `/dev/shm/*.log` (例: `yata_task.log`, `yata_dashboard.log`, `yata.log`, `api_usage.log`)
*   **用途**: Cronタスクや `run-ram.sh` 経由で1日中頻繁に書き込まれる、日常の運用ログやAPIの個別通信記録(`api_usage.log`)。
*   **特徴**: すべてメモリ上に書き込まれるため、**SDカードの寿命を1秒たりとも削りません。** (`iostat` にて日中のSD書き込みがほぼゼロであることを実証済み)。
*   **運用**: 毎朝のバックアップ後に `truncate -s 0` で空にされ、新しい24時間の記録が始まります。

#### 2. 長期・アーカイブ保存（NASへの日次退避）
*   **場所**: `/mnt/nas/yata_logs/`
*   **用途**: トラブル発生時の過去履歴の調査用. 特に `api_usage_YYYYMMDD.log` はAPIコスト異常高騰時の原因特定（モデルごとの詳細なトークン消費記録）に利用。
*   **特徴**: 毎朝4:35のバックアップ実行時に、RAM上にあるその日のログファイルをまるごとNASへ日付付き (`*_YYYYMMDD.log`) でコピーします。コピー後、RAMのログは直ちにリセットされます。NAS上では過去30日分が保存されます。

### 過去記事の構造化バックフィル (2026/03/26 導入)
*   **ツール名**: `toolBackfillStructuredSummaries(totalLimit, batchSize)`
*   **目的**: 構造化（5W1H JSON化）導入前の過去記事を、一括で再要約・構造化し、AI分析の精度を底上げします。
*   **挙動**: E列（Summary）が `{` で始まっていない記事を「プレーンテキスト」と判定し、5件ずつのバッチ（`batchSize=5`）で一括再要約を行います。
*   **実行方法**: GAS エディタの関数一覧から選択して手動実行。`totalLimit`（デフォルト50）で処理件数を調整可能。

### 📝 Strategic Memo: 構造化JSON（5W1H）導入の真の意義と将来構想
*   **「焼き上がったケーキ」から「素材の冷蔵庫」へ**:
    *   従来は記事を「要約されたテキスト」として保存していたが、これを「5W1H＋KeywordsのフルJSON」としてDBに直保存する設計に移行。
    *   これにより、1回のLLM処理で得た「要素（who, what, how 等）」を、将来のあらゆる分析や可視化で「再推論のコストゼロ」で使い回せるようになった。
*   **抽出の二度手間の完全排除（ベクトル生成の進化構想）**:
    *   新JSONの `what`（技術）と `how`（手法）を結合すれば、それがそのまま最強の「手法ベクトル用素材」になる。
    *   将来的に、ベクトル生成（Embedding）の際、生のテキストを読ませるのではなく、**DBに眠るJSONの該当要素だけをパース・結合してAPIに渡す**仕様に進化させる。これにより、APIコストと処理時間を極限まで削ぎ落とす。

### 📝 戦訓：1.4.0 大規模統合時のカラムマッピング不整合 (2026/03/28)
- **事象**: `YATA.js` 1.4.0 移行に伴う 17 カラム構造化の際、SQLite 保存ロジック (`gas-bridge.js`) で引数の順序がズレ、ID カラムにタイトルが混入したり、記事が空白になる致命的な不整合が発生した。
- **原因**: 1.2.5 以降の列順変更に対し、`stmt.run()` に渡す引数のインデックス指定を物理的に一行ずつ突き合わせる検証を怠ったこと（推測による OK 報告）。
- **解決**: 
    - `lib/gas-bridge.js` の保存ロジックを再編し、引数順序を SQLite 定義に完璧に一致させた。
    - 不正な ID（URL 以外）を持つデータを DB から一掃。
    - **健全性チェック・スクリプト (`tests/verify-db-integrity.js`)** を作成し、物理的に不整合が 0 件であることを確認するフローを確立。
- **教訓**: 大規模なデータ構造変更の際は、「件数」だけでなく「中身（各カラムの妥当性）」を必ず `SELECT` して目視確認すること。

## 4. 環境設定 (.env) のベストプラクティス

### データベースパスの固定 (2026/03 強化)
SDカードの摩耗を物理的に防ぎ、常に最高のパフォーマンスを発揮するため、`.env` 内の `DB_PATH` は以下のように **RAM ディスクを指す設定** が標準となりました。

```env
DB_PATH=/dev/shm/yata.db
```

*   **効果**: `run-ram.sh` を介さずに実行される Node プロセスも、デフォルトで RAM 上の DB を読み書きするため、誤って SD カードを直接叩くリスクがなくなります。
*   **整合性**: `run-ram.sh` が終了時に SD カードの実体 (`~/yata-local/yata.db`) へ物理的に書き戻すため、永続性は保たれます。

### AI推論パラメータの完全外部化 (GPT-5.4対応 / 2026/03)
最新の `gpt-5.4` モデルへの移行に伴い、`lib/YATA.js`（本家と共通の聖域コード）を一切汚すことなく、ローカル環境（.env）および会社環境（GASのスクリプトプロパティ）からモデルの挙動を直接制御できる「完全外部化」を実現しています。
※ OpenAIの厳格な仕様により、5.4系推論モデルでは `temperature` および `top_p` の両方が使用不可（エラー）となるため、システム内部で自動的に送信を除外する安全設計となっています。

**▼ 推奨される `.env` 設定（ローカルRasPi用）**
```env
# 1. 蒸留圧縮フェーズ（高密度化: 不要な修辞を削ぎ落とす）
OPENAI_MODEL_NANO="gpt-5.4-nano"
VERBOSITY_NANO="low"
REASONING_NANO="none"

# 2. 統合分析フェーズ（インサイト抽出: 文脈を読み解く）
OPENAI_MODEL_MINI="gpt-5.4-mini"
VERBOSITY_MINI="medium"
REASONING_MINI="high"
```

**【会社（GAS）環境での運用】**
これらのプロパティはすべて `|| "minimal"` などの既定値（フォールバック）を持っているため、**会社側のGAS設定（スクリプトプロパティ）に何も追加しなくても、自動的に従来の安全な設定で稼働します。** 
将来、会社環境でも新モデルへ移行したくなった時だけ、GASエディタの「プロジェクトの設定 > スクリプト プロパティ」に上記と同じキーと値を追加するだけで、コード変更ゼロで即座に切り替えが可能です。

### 軽量化フック (爆速メンテナンス)
`lib/gas-bridge.js` には、1.4万行を超える DB でも収集ジョブを遅延させないための「SQL フック」が実装されています。35日以上前のベクトルデータは、収集ジョブの開始時に数ミリ秒で自動消去され、DB サイズの健全性が保たれます。

#### 3. 致命的エラー用の遺言ログ（SDカード直書き）
*   **場所**: `/home/boncoli/yata-local/backup.log`
*   **用途**: `do-backup.sh` 自体の実行結果を記録。
*   **特徴**: バックアップ処理やNASへのマウントが失敗した際、RAMやNASにログを頼っていると「失敗したこと自体が記録できない」というデッドロックを防ぐための**唯一の例外**です。1日1回数行しか書かれないため、SDカードへの負荷は無視できます。

> 💡 **トラブル時の調査フロー**:
> 1. 今日のエラーを見る → `cat /dev/shm/yata_task.log` などを確認。
> 2. 昨晩以前のエラーを見る → `/mnt/nas/yata_logs/` を確認。
> 3. 朝起きてシステムが死んでいた → `/home/boncoli/yata-local/backup.log` を確認。

#### 災害復旧 (Disaster Recovery) 手順 (2026/03/25 更新)
万が一 SD カードが全損した場合は、以下の手順で復旧を試みます。
1.  **OS基盤**: 新しい SD カードに Raspberry Pi OS をインストールし、最低限のネットワーク設定を完了させる。
2.  **パッケージ復元**: バックアップされた `package_list.txt` を参照し、必要なパッケージを再インストールする。
3.  **マウント復元**: `/etc/fstab` を確認し、NAS マウント (`/mnt/nas`) を再構築する。
4.  **データ連戻し**: NAS の `home_backup` からホームディレクトリの内容を `rsync` で書き戻す。
5.  **スケジュール復元**: `crontab crontab_last.txt` を実行して cron 設定を復元する。
6.  **インフラ再構築 (重要)**: 
    - `yata.db` を `444` (Read-Only) にロックして物理ガードを施錠する。
    - メモリDBへのリンク (`yata-mem.db`) およびダッシュボード画像へのリンク (`dashboard.png`) を再構築する。
    - Tailscale のログイン状態とサービスバージョンを確認する。
7.  **サービス起動**: `pm2` 等の常駐プロセスを起動し、ポータルのメイドさん（画像アセット）が表示されているか確認する。

### 設定データのローカル管理 (2026/02 移行)
システムの自律性を高めるため、外部（Google スプレッドシート等）に依存していた設定データをローカルの JSON ファイルへ移行しました。

*   **RSSフィード**: `rss-list.json` (プロジェクトルート)
*   **LLMプロンプト**: `prompts.json` (プロジェクトルート)
*   **優先順位**: `gas-bridge.js` は、これらの JSON ファイルが存在する場合、スプレッドシートよりも優先して読み込みます。
*   **管理**: フィードは `tasks/manage-feeds.js` で管理可能です。プロンプトは直接 JSON を編集するか、後述の同期コマンドでスプレッドシートから更新できます。

### 通知・レポート機能 (Discord 4チャンネル運用)
情報の視認性向上と整理のため、Discord通知を内容ごとに4つの専用チャンネルに分類して運用しています。
*   **`#yata-digest`**: 毎朝10:02のパーソナライズされたニュース・ハイライト。
*   **`#drive-log`**: 車の乗り降り、走行距離計算結果、給油・燃費報告。
*   **`#system-alerts`**: バックアップの開始・完了通知、および重大なシステム異常アラート。
*   **`#ai-mutter`**: AI（Gemini）による自律的な独り言（5分〜1時間おき）。

*   **リアルタイム通知 (2026/02 強化)**: 
    - `OutCar` 時: 今回の推定走行距離を自動計算して `#drive-log` に投稿。
    - 給油時: 今回の区間燃費を即座に計算して `#drive-log` に報告。

### SDカード保護プロトコル (最強の物理ガード) (2026/03/16 導入)
SDカードの摩耗を極限まで抑え、かつ誤操作による直接書き込みを物理的に遮断する仕組みです。

#### 1. OSレベルの Read-Only ロック
- **SD上の実体DB (`yata.db`) は、常時 `chmod 444` (読み取り専用) に設定されています。**
- これにより、どんな言語やスクリプトから直接アクセスしても、OSレベルで書き込みが拒否されます。

#### 2. run-ram.sh による自動開閉
- すべてのDB操作（書き込み・更新）は必ず `bash run-ram.sh <script.js>` を経由します。
- **実行フロー**:
    1. **展開**: SDからRAM (`/dev/shm/yata.db`) へDBをコピー。
    2. **実行**: 環境変数 `DB_PATH` をRAM上のパスに書き換えてスクリプトを実行。
    3. **解錠**: 終了後、`run-ram.sh` が一瞬だけSDの権限を開放 (`chmod 666`)。
    4. **同期**: RAMからSDへ安全に書き戻す。この際、WALファイルの亡霊化を防ぐため、RAM側に `-wal` がない場合はSD側の `-wal` も自動削除されます。
    5. **施錠**: 書き戻し完了直後、再び実体DBを `chmod 444` にロックします。

#### 3. 開発・メンテナンス時の注意
- **直接SQLを叩く場合**: `sqlite3 /dev/shm/yata.db` (RAM側) を使用してください。
- **直接実行の禁止**: `node tasks/...` などの直接実行は、OSによって弾かれます。これは「正常な防壁」です。
- **例外**: どうしても直接書き込みたい場合のみ、一時的に `chmod 666` するか `ALLOW_DIRECT_DB=1` を使用しますが、原則として非推奨です。

## 10. システム安定化・高速化・時限爆弾の解体記録 (2026/02/25)

### DB二重起動問題の解消
- `server.js` と `gas-bridge.js` のDB接続を一本化し、`malformed` エラーを根絶。

### 地図データの超軽量化
- `japan.geojson` を 13MB -> 0.65MB (95%削減) に軽量化。Gzip 圧縮とキャッシュ有効化により爆速化を実現。

### エレベーター対策ロジック
- マンションのエレベーター等による 5分以内の一時的な Wi-Fi 切断を「外出」とみなさないバッファロジックを実装. 

### OOM枯渇とOSフリーズ問題の完全解消 (2026/03/06)
- **原因**: `gas-bridge.js` において、GASの同期通信をNode.jsでエミュレートするために使用していた `sync-fetch` が、通信のたびに新たなNodeワーカープロセスを起動していた。これが大量のRSS処理やCronの重複と相まってメモリを急激に食い潰し、ラズパイのOSフリーズ（および書き込み中の `yata.db` のデータ消失）を引き起こした。
- **対策**: `sync-fetch` ライブラリを完全撤去. 代わりにLinux標準の `curl` コマンドを `child_process.execSync` で呼び出す方式へ書き換えたことで、プロセス乱造とメモリ肥大化を根絶し、超軽量・安全な同期通信を実現した。

### 潜在的な時限爆弾の解体と予防措置 (2026/03/06)
1. **PM2ログ肥大化爆弾（SDカード摩耗対策）**:
    - `pm2-logrotate` を導入。各ログを最大10MBに制限し、7日分だけ保持して古いものを自動圧縮・削除する設定を適用。無限のログ蓄積によるSDカード寿命の低下を防止。
2. **AIの熱暴走・スロットリング爆弾（フェイルセーフ）**:
    - ローカルLLM（Gemma 3）が1時間おきに実行される `tasks/do-local-mutter.py` の冒頭に、ラズパイのCPU温度チェックを実装。70℃を超えている場合はAIの起動をスキップし、熱暴走とOSフリーズを未然に防ぐ。
3. **不要な書き込みとコアダンプ（ゴミデータ）爆弾**:
    - `tasks/process-mutter.js` で Node.js v24 の `fetch` 直後の終了によるクラッシュ（85MBの `core` ファイル大量生成）が発生していたため、安定した `https` モジュールへ書き換えて根絶。
    - 定期タスク（`yata-task` 等）の重いログ出力をSDカードからRAMディスク (`/var/log/`) へ逃がし、毎朝のバックアップ時にRAM上の全ゴミ（一時ファイル等）を強制掃除するロジックを確立。

## 11. 公開リポジトリ (YATA-GAS) の分離運用とサニタイズの鉄則 (2026/03/08 確定)

2026年3月8日、GAS本家版コアロジックを「YATA-GAS」としてGitHub上でOSS公開した。秘密情報の流出防止とリポジトリの純粋性維持のため、**「3層ブランチ構成」**による分離運用を「絶対の掟」とする. 

### 11.1 3層ブランチ構成と役割
ブランチ名を分けることで、プッシュ先の間違いを物理的に防ぐ。

| ブランチ名 | 対応リモート | ファイル構成 | 役割 |
| :--- | :--- | :---: | :--- |
| **`local-raspi`**| `origin/local-raspi` (Private: `YATA`) | 全ファイル | **開発・運用ブランチ。プロジェクトの「唯一の正本 (SSoT)」。** |
| **`main`** | `origin/main` (Private: `YATA`) | **6ファイル** | **会社共有用。** 会社環境（GAS等）で必要なコアセット。 |
| **`public`** | `public/main` (Public: `YATA-GAS`) | **6ファイル** | **OSS公開用。** 機密情報を除いた配布用。履歴は常にスカッシュ。 |

### 11.2 公開・同期時の絶対手順
各ブランチへの反映は、必ず `local-raspi` での動作確認・ドキュメント更新後に行うこと. 

1.  **`main` への同期**: `lib/YATA.js` 等のコアファイルを `main` へ反映。会社用プロンプト (`prompt_company.json`) を含める。
2.  **`public` への同期**: `lib/YATA.js` をサニタイズ（機密情報置換）して反映。公開用プロンプト (`prompts.json`) を使用。
3.  **歴史の管理**: `public` ブランチは履歴を 1 つのコミットにまとめる（スカッシュ）。

### 11.3 構成ファイル・マトリックス (2026/03 整理済)

### 11.5 OSS版のバージョン管理と GitHub Release フロー (v1.0.0 以降)
v1.0.0 (Modern Era) の公開以降、OSSとしての健全なバージョン管理と機密情報の保護を両立するため、以下のフローでリリースを行う。

1. **機密情報の完全排除 (サニタイズ)**: `.env` などの個人的な設定ファイル、および `lib/YATA.js` 内の個人特定情報（IPアドレス等）が含まれていないことを徹底的に確認する。
2. **履歴のリセットとクリーンコミット**: 過去の開発履歴（試行錯誤の過程にある設定値の漏洩リスク）を断ち切るため、公開用ブランチは常に `git checkout --orphan` で履歴をリセットし、対象ファイル（上記マトリックス＋ `prompt_template.json` 等）のみを追加して「1つのクリーンなコミット」を作成する。

4. **強制プッシュと Release 作成**: 
   - `git push public <branch>:main -f` で OSS リポジトリを上書き更新する。
   - `git push public <tag> -f` でタグをプッシュする。
   - GitHub 上の `YATA-GAS` リポジトリページで「Releases」から当該タグを選択し、リリースノート（変更点）を記載して Publish する。これにより、ユーザーに対して「安全で明確なバージョンアップ」を提供できる。

    - **CPU (鉄壁の保護)**: `n_threads=3` 指定。さらに **`taskset -c 0-2` により 4コアのうち 1コア（コア3）をシステム専用に物理隔離。** AI推論中もCronやDB同期を滞らせない. 
    - **優先度**: `nice -n 15` でバックグラウンド実行を徹底。

### 14.2 Discord 連携 (yata-bot)
- **通信方式**: `child_process.execSync` (CLI) から `fetch` (HTTP API) へ移行。
- **高速化 (Caching)**: システムプロンプトの構成を最適化（固定ペルソナを冒頭に、変動コンテキストを末尾に配置）。Llama.cpp の KV キャッシュを最大限活用し、生成開始までのレイテンシを極限まで削減. 
- **コンテキスト統合**: 返信時に以下の情報をプロンプトへ自動注入。
    - **位置情報**: `drive_logs` (在宅、外出、ドライブ中、目的地)
    - **環境情報**: `weather_log` (天気、気温)
    - **自己診断**: CPU 温度 (自身の「体調」として言及可能)
- **フェイルセーフ**: 
    - **熱暴走防止**: CPU 75℃ 超過時にローカル推論を自動スキップ。
    - **クラウド・フォールバック**: ローカルサーバー無応答時、即座に OpenAI (`gpt-5-nano`) または Gemini に切り替えて返信を継続。

### 14.3 運用のメリット
- **レスポンス**: ロード時間 (20秒) が消失。数秒での返答が可能。
- **プライバシー**: 高度な個人情報（位置履歴等）を含む推論を、外部サーバーへ一切送信せずに完結. 
- **レジリエンス**: `@reboot` 指定の `run-ram.sh` により、停電復旧後も自動で DB 展開・サーバー起動が完了する.

## 15. 技術調査メモ：OpenAI Responses API & Data Share 無料枠 (2026/03/23)

OpenAI が導入した「データ共有同意による無料（または低価格）枠」の活用を検討したが、現在の YATA のリアルタイム要約運用には「不適合」と判断し、従来の Chat Completions API に差し戻した。

### 15.1 調査結果：なぜ実用性が低いのか
1. **極端なレイテンシ（速度不足）**:
   - `/v1/responses` エンドポイントは非同期（Asynchronous）専用。
   - リクエスト後、OpenAI 側のキューに積まれ、生成開始まで数秒〜数十秒の待機が発生する。
   - GAS 側で `Polling`（終わったか何度も確認する処理）が必要になり、合計で 1 分以上の待ち時間が発生することがザラにある. 
2. **無料枠のトレードオフ**:
   - 「Data Share（無料）」を有効にするには、`store: true`（履歴保存）が必要。
   - しかし、`store: false` を指定しないと生成速度（レイテンシ）がさらに悪化するというジレンマがある. 
3. **不安定なステータス**:
   - 混雑時、`status: incomplete` が延々と続き、最終的にタイムアウトするリスクが高い。

### 15.3 OpenAI Responses API 統合 (2026-03-24 実装)
- **エンドポイント**: `https://api.openai.com/v1/responses` を採用。
- **特徴**: 従来の Chat API よりも最新の推論モデル（gpt-5, o1等）に最適化された非同期ライクな同期通信。
- **高速化オプション**: `reasoning_effort: "low"` を標準適用。これにより、gpt-5-nano での要約速度が **約 1/6 (60s -> 10s)** に短縮. 
- **コスト計算**: `usage` オブジェクトから正確なトークン数を取得し、`_recordUsage` へ連動。

### 15.4 ID ベースのバッチ要約パッチ (Enhanced Batch Distillation)
- **バッチサイズ**: 5記事一括。
- **ID 紐付け**: 各記事に `id: "0"～"4"` を付与してパッキング。返却された JSON 配列から ID をキーに元の順序へ復元。
- **メリット**: 順序保証のない Responses API 下でもデータの整合性を 100% 維持。
- **フォールバック**: JSON 崩壊やタイムアウト時は、自動的に 1記事ずつの個別処理へ切り替わる二重安全構造。

## 16. マルチプラットフォーム運用コンソール (Mac/Windows 対応)

ラズパイ実機を直接操作せず、手元の PC (Mac/Windows) から SSH 経由でメニューを操作し、必要に応じてローカルブラウザや別窓ターミナルを自動起動する「ハイブリッド運用環境」を構築しています。

### 16.1 アーキテクチャ (v5.4.0 Session-Isolated)
*   **サーバー側**: `yata-menu-core.sh` (Bash)
    - 22個のコマンドを集約。
    - **セッション隔離**: 起動時にランダムなセッションIDを受け取り、`/dev/shm/yata-console.[ID].signal` という専用の通信路を確立。これにより多重起動時の誤作動（ウィンドウが複数開く等）を物理的に排除。
*   **クライアント側 (ランチャー)**:
    - **Mac**: `local_public/yata-launcher-mac.txt`
    - **Windows**: `local_public/yata-launcher-windows.txt`
    - **低遅延通信**: SSH をパイプから解放し、`-t` によるインタラクティブ接続を採用。入力遅延をゼロ化。
    - **バックグラウンド監視**: メインセッションの裏で `tail -n 0 -f` を使い、自分専用の信号ファイルを監視。命令検知時に Windows/Mac 側のネイティブ機能を叩く。

### 16.2 実装の急所 (2026/03/24 革命)
*   **入力遅延の完全解消**: パイプによるバッファリング問題を SSH 生接続により根絶。ラズパイ実機同等のレスポンスを実現。
*   **別窓ターミナル (8番)**: 
    - **Windows**: PowerShell の `Start-Process` を使用し、独立したウィンドウで SSH セッションを開始。
    - **Mac**: AppleScript (`osascript`) を使用し、Terminal.app で新しいウィンドウを起動。
*   **セッション・クリーニング**: 終了時に `trap` 命令によりバックグラウンド監視を確実に殺し、ラズパイ側の信号ファイルも削除する鉄壁の後始末を実装。

## 17. 検証・テストの黄金プロトコル (2026/03/24 策定)
大規模な AI 機能（要約・バッチ・ベクトル生成）の修正後は、必ず以下の「黄金のテストセット」を実行し、物理的な動作証明を行うこと。

- **目的**: 複数記事（5件等）を同時に投げた際、ID (0, 1, 2...) の紐付けが崩れないかの確認。
- **合格基準**: 各記事に対して正しい要約が配列に再配置され、「真っ白」なスロットが存在しないこと。

### 17.3 実データ統合検証 (The Gold Test)
- **コマンド**: `node tests/test-ai-integrated.js`
- **目的**: 実DB (`/dev/shm/yata.db`) の記事を用い、要約から256次元ベクトル生成までの一気通貫の成功。
- **合格基準**: 全記事（5件）の要約成功と、それぞれに対する256次元ベクトルの生成成功。
- **重要**: `_callOpenAiEmbedding` が欠落していないことを物理的に証明する唯一のテスト。

### 高密度コンテキスト・エンジン (High-Density Context Optimization)
- **概要**: 構造化抽出された 5W1H (who, what, how, result 等) の JSON を「素材」として扱い、分析フェーズで AI に渡す直前に論理的順序で再構成する。
- **最適化ロジック**: 
    1. 構造化 JSON から `Unknown`（情報なし）の要素を物理的に排除。
    2. **WHAT → HOW → RESULT → WHO → WHY → WHEN → WHERE → TLDR → KEYWORDS** の順で結合。
    3. JSON の記号（`{}` や `""`）を剥ぎ取り、純粋なテキスト素材のみを提示。
- **導入効果**: 
    - **トークン削減**: 文字数ベースで約 16〜30% の削減（実測）。
    - **精度の向上**: AI が「手法（HOW）」や「成果（RESULT）」の共通性をよりダイレクトに捉えられるようになり、トレンド分析の質が向上。
    - **スケーラビリティ**: 同じトークン制限内で、より多くの記事（より広い文脈）を一度に分析可能になった。
